# Передача работы: сквозная аналитика «Прознание / CODDY»

Состояние на 23.07.2026 (после полудня). Репозиторий: `Jane3dMp/fb_dashboard`,
ветка `main`. Развёрнута **версия 29** Apps Script (11:02), идентификатор
развёртывания прежний.

## Цель

Ответить на вопрос «окупается ли реклама в Meta» и в каком направлении.
Цепочка: расход Meta → заявка в amoCRM → оплата в AlfaCRM, с разрезом
по направлениям (каникулы отдельно от CODDY).

## Главное: мост починен, деньги настоящие

На окне 90 дней (24.04–23.07) страница «Путь клиента» показывает:
расход в Instagram **$3 117**, оплативших **103**, выручка **59 502 BYN**,
стоимость клиента **88 BYN**, **ROAS 6.58×** (по курсу FX_RATE 2.9).
По направлениям: Уикенд 50 122 · CODDY+Прознание 8 250 · Детали 1 130.

Это **нижняя граница**: с Альфой связано 12% заявок (146 из 1219) — у 78%
заявок из Instagram в amo нет телефона (переписка без обмена номером).
Из заявок С телефоном находятся в Альфе 54%.

**Как устроен мост** (`pplAlfaRevenueCore_` в `apps-script/people.gs`,
чистое ядро, гоняется в Node):
1. `alfa_url` из карточки amo — пока пуст у всех, но если начнут заполнять,
   подхватится сам и имеет приоритет;
2. телефон: `phone_e164` заявки против нормализованных номеров клиента
   Альфы из листа `RAW_alfa_customers`. Нормализация одинаковая с двух
   сторон (`pplNormPhone_` = `normalizePhone_` из etl_amo).
Семья (несколько детей на один номер) считается одним оплатившим,
деньги каждого клиента — один раз, платежи только с даты заявки.

**Проверено тройной сверкой**: локальная симуляция на боевых выгрузках
(`node simulate.js` в scratchpad прошлой сессии) → живой JSON → страница.
Цифры совпали до копейки (59 501.50).

## Что вскрылось и починено по дороге

**Старая выгрузка платежей была сломана трижды** (файл `Etl alpha.gs`,
больше НЕ используется, триггеры сняты):
1. Фильтр дат передавался как `document_date_from/to` — Альфа эти имена
   молча игнорирует (рабочие: `date_from`/`date_to`). Каждый «месяц»
   тянул ВСЮ историю с 2022 года.
2. Отметка месяца в `RAW_pays_meta` сравнивала строку '2026-07' с Date
   из ячейки — никогда не совпадала, текущий месяц грузился заново каждый
   день. Итог: в RAW_pays лежало 76.6 тыс. строк — история в ~3 экземплярах.
3. `ALFA_BRANCHES` был '1,2' — платежи Каникул (филиал 4, 867 шт.) и
   Интенсивов (филиал 5, 72 шт.) не выгружались вовсе. Свойство теперь
   **'1,2,4,5'**.

**Филиал 3 («Якубовского, 90») — зеркало филиала 1**: те же клиенты и те
же платежи с теми же id (проверено по API). В ALFA_BRANCHES его добавлять
НЕЛЬЗЯ — всё задвоится. Дедуп по pay_id в коде на этот случай есть.

**Новая схема платежей**: `pplRebuildPays()` (в people.gs) ежедневно
пересобирает лист RAW_pays целиком: только доходы (`pay_type_id: 1`
серверный фильтр + страховка в коде), дедуп по pay_id, защита от пустого
ответа (< 1000 платежей — лист не трогается). Сейчас в листе **28 730**
строк — ровно столько, сколько отдаёт API. Прогон занимает ~6 минут — на
грани лимита; если начнёт падать по таймауту, разнести филиалы по двум
триггерам (1 отдельно, остальные вместе).

**Новый лист `RAW_alfa_customers`** (заполняет `pplEtlAlfaCustomers()`,
1651 клиент): customer_id, branches, phones (нормализованные, через `;`),
amo_contact_id (из стандартного поля `web` Альфы — заполнен у ~18%,
пригодится для точной сверки, если в RAW_leads появятся id контактов),
created_at.

**Мёртвые поля, на которые не надеяться** (проверено по API):
поле контакта amo «(X) Ссылка на alfaCRM» (id 1652617) пусто у всех
заявок; обратное поле Альфы `custom_url_amo_client` пусто у всех клиентов.

## Решения, которые лучше не пересматривать без причины

- **Платежи считаются с даты заявки**; период — по дате заявки, не
  платежа; один клиент один раз; доля связанных на странице. Всё как было.
- **Полный пересбор RAW_pays вместо инкремента** — дёшево (6 мин/день),
  самовосстанавливается, нет учёта месяцев, который уже один раз наврал.
- **Хелперы people.gs с префиксом ppl** — общая глобальная область,
  коллизии имён уже случались.

## Как деплоить (проверенный путь без мучений с буфером)

1. Правки локально → коммит → `git push` (repo = источник и для Pages).
2. В редакторе Apps Script (вкладка Chrome с аккаунтом proznanie):
   через javascript в консоли страницы записать файл прямо в Monaco:
   модель people.gs находится по `monaco.editor.getModels()` (798→1021
   строк была её длина), `fetch` raw.githubusercontent + `setValue`.
   Затем Ctrl+S, проверить «Сохранено на Диске» и что в списке функций
   появились новые.
3. «Начать развертывание → Управление развертываниями → карандаш →
   Версия: Новая → Развернуть» — id развёртывания сохраняется.
4. Проверка: `/exec?view=people&days=90&nocache=1&key=<DASH_KEY>`.
Капризы редактора (залипание списка функций, срыв вставки из буфера)
описаны в старых заметках ниже — с Monaco-способом они не мешают.
